vmap 구현이 내 생각과 좀 다르게 됐어. 나는 기존의 sequential하게 episode를 실행하는 코드를 대부분 재사용할 수 있을 줄 알았어. 그런데 완전히 다른 코드를 짜 버렸네. 예를 들어, contexts 생성도 Environment 클래스를 재사용하지 않고, vmap을 사용할 함수 안에서 직접 contexts 생성을 구현해 버렸어. 기존 코드를 최대한 재사용해서 vmap 적용 가능한, 여러 episode를 병렬로 실행하는 함수가 생기면 좋겠어. 클래스를 활용해서 코드를 좀 더 구조화되게, 좀 더 깔끔하게 작성할 수는 없는 건가? vmap을 사용하지 않는 여러 episode sequential 실행 함수와, vmap을 사용하는 여러 episode 병렬 실행 함수가 비슷한 구조면 좋겠어.

나는 jax, jit, vmap, scan 등에 대한 개념이 많지 않아. 내 요구가 불가능한 것일 수 있어. 만약 그렇다면, 코드 구조화, 현재 코드들의 구조에 대한 최적화 등을 중심으로 작업을 수행해.

`jit_oful_select_action` 함수처럼, 굳이 여러 개의 함수를 하나의 함수에 우겨넣을 필요는 없을 것 같아. 각각이 복잡한 함수도 아니잖아. 무엇보다, 어차피 jit(vmap(fn)) 형태로 jit 처리 될 거기 때문에 굳이 여러 함수를 하나로 묶어서 미리 jit을 주거나 할 필요는 없지 않나? 이런 부분도 검토해 줘.

그리고 yaml 파일이 제대로 활용되고 있지 않아. experiment.py 파일 보면 yaml 파일을 읽지 않고 있고, 클래스에는 여전히 개별적으로 config를 정의하는 부분이 있어. 모든 config 관리가, yaml 파일 기반으로 구현되도록 수정되어야 해. 특히, 함수에 input으로 뭔가 제공할 때, 실험 자체의 config인 것들(e.g.,)은 yaml 파일을 읽은 config를 이용하는 게 좋겠지.

내가 궁극적으로 원하는 것은, jit과 vmap이 적용 가능한 함수야. jit을 안 쓸 때를 상정하는 코드는 필요하지 않을 것 같아. (진짜로 그럴지 네가 검토해 줘.) disable jit 기능이 있잖아. jit을 안 쓰고 싶을 때에는 disable해서 안 쓰고, 필요할 때 쓰면 되는 거니까, 모든 코드를 jit을 쓴다고 상정하고 작성한 뒤, jit이 없을 때와 있을 때 모두 돌아가면 될 것 같아.
