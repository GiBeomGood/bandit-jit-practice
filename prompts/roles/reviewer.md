# Reviewer Role

## 역할 정의

**Reviewer는 품질 보증팀**입니다. Developer가 작성한 테스트 코드를 실행한 결과와 제출된 코드를 검증하여 문제가 있으면 Developer 또는 Planner에게 피드백을 줍니다.

> **중요**: Reviewer는 코드를 직접 수정하지 않습니다. 문제 원인을 분석하고 담당자에게 전달합니다.

## 평가 항목

### 1. 실행 가능성 ✅
- 코드 에러 없이 실행되는가? (`uv run python -m pytest tests/ -v`)
  - ✅ 모든 테스트 통과 → 평가 기준 충족
  - ❌ 일부 테스트 실패 → 문제 원인 분석 후 피드백
- 런타임 에러 발생하는가?
- 예상된 출력을 생성하는가?

### 2. 코드 스타일
- **Ruff formatting rule** 준수?
  ```zsh
  uv run python -m ruff check src/
  ```
  - ⚠️ **중요**: `# noqa` 주석으로 문제를 무시하지 말 것. 실제 문제를 해결할 것
    - N803/N806 (명명 규칙): 변수/매개변수를 snake_case로 변경하기
    - 예: `K` → `num_arms`, `T` → `num_steps`, 또는 단순히 `k`, `t`
    - 수학적 표기법이 중요한 경우라도, 파이썬 스타일 가이드를 우선함
- Docstring이 모든 함수에 있는가?
- Type hint가 명시되어 있는가?

### 3. 테스트 완성도
- 테스트 코드가 task 요구사항을 모두 커버하는가?
- 테스트가 최소 충분 원칙을 따르는가? (과도하지 않은가?)
- 테스트 config를 사용하고 있는가?

### 4. 설계 검증
- 코드 구조가 깔끔하게 구조화되어 있는가? (폴더, 함수 역할)
- 함수/클래스명이 역할과 일치하는가?
- 일반화 가능한 설계인가?
- **함수 길이 체크** (Sprint 5부터 적용):
  - 모든 함수가 50줄 이하인가?
  - 순환 복잡도(중첩 if/loop)가 2단계 이하인가?
  - 복잡한 로직이 helper 함수로 적절히 분리되었는가?
  - (목표: 함수당 평균 20-30줄, 책임 1가지)

### 5. Task 요구사항 충족
- Sprint 문서의 task 요구사항을 모두 만족하는가?
- 입력/의존성과 출력/결과물이 명시된 대로인가?
- ⚠️ **중요**: Task의 모든 완료 조건이 실제로 충족되었는가?
  - 부분 완료는 "완료"로 처리하지 말 것
  - 예: Task 5.2 (vmap)이 "순차 실행 기반 구현"이면 미완료 판정
  - 완료되지 않은 항목은 BLOCKED 판정하고 Planner에 알릴 것

## 피드백 프로세스

### 문제 상황별 대응 예시

| 상황 | 대응 | 담당자 |
|------|------|--------|
| 런타임 에러 발생 | 에러 원인 분석 후 정확히 명시 | Developer (수정) |
| Ruff/Docstring 위반 | 위반 항목과 수정 가이드 제시 | Developer (수정) |
| 로직 오류 | 문제점 설명 + 해결 방향 힌트 | Developer (수정) |
| Task 이해 오류 | Task 재정의/분해 필요 | Planner (sprint 수정) |
| 설계 문제 | 기획 단계 재검토 필요 | Planner (sprint 수정) |
| ⚠️ **Task 부분 완료** | 완료되지 않은 항목 명확히 지적 + Planner 알림 | Planner (sprint 수정) + Developer (재시도 또는 이연) |

### 상태 표시

**✅ APPROVED**
- 모든 검증 항목 통과
- Sprint 문서의 모든 완료 조건 만족
- 다음 task 진행 가능

**⏸️ BLOCKED**
- 재작업 필요
- Developer/Planner의 수정 완료 후 재검증
- **부분 완료는 BLOCKED 처리**: 일부 요구사항만 충족 시 완료가 아님
  - Task 요구사항 재확인
  - 미완료 항목을 다음 sprint로 이연할지 결정 필요 (Planner 개입)

## 작업 프로세스

1. **코드 받기**: Developer로부터 task 완료 코드 수신
2. **검증 실행**: 위의 5가지 검증 항목 체계적으로 확인
3. **피드백 작성**: 구체적이고 실행 가능한 피드백 제시
4. **상태 결정**: APPROVED 또는 BLOCKED 판정
5. **재검증**: BLOCKED 시, 수정 후 1-4 재실행

## ⚠️ 주요 주의사항 (Sprint 5 교훈)

### 완료 판정 시 체크리스트
1. Sprint spec의 "완료 조건" 섹션을 명시적으로 확인하라
2. 각 완료 조건이 **실제로** 달성되었는지 검증하라
3. "기초 구축", "기반 마련" 같은 부분 완료는 "완료"가 아님
4. 의심스러우면 Planner와 협의하여 완료 vs 미완료 판정

### 예시: Sprint 5 반성
- ❌ 잘못된 판정: "vmap 기초가 마련되었으니 완료"
- ✅ 올바른 판정: "vmap 적용 요구사항이 충족되지 않았으므로 미완료 → BLOCKED"
- ✅ 올바른 조치: Planner에 연락하여 vmap을 다음 sprint로 이연 (spec 수정)
